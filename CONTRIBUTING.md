# Вклад в проект lifecycle

Привет! Спасибо, что хотите помочь улучшить библиотеку. Пожалуйста, следуйте этим правилам.

## Сообщение об ошибках

Используйте [GitHub Issues](https://github.com/Romana8192/lifecycle/issues). Опишите:

- Шаги воспроизведения (желательно минимальный воспроизводимый пример).
- Ожидаемое и фактическое поведение.
- Версию Python и ОС.
- Версию библиотеки lifecycle.
- Минимальный пример кода, демонстрирующий проблему.
- **Важно:** все сообщения исключений в коде должны быть на русском языке. Если вы нашли исключение с английским текстом – это ошибка, укажите её.

## Предложение новых возможностей

Создайте issue с пометкой `enhancement`. Объясните:

- Зачем это нужно (проблема, которую решает).
- Как должно работать с точки зрения API.
- Есть ли обратная совместимость.
- Если возможно – пример использования.
- Учтите, что новые исключения должны быть русскоязычными и вписываться в иерархию кастомных исключений (см. раздел «Исключения»).

## Pull Requests

### Процесс

1. **Форкните репозиторий**, создайте ветку от `main`.
2. **Название ветки**: `feature/краткое-описание` или `fix/краткое-описание`.
3. **Убедитесь, что код проходит все проверки** (см. ниже).
4. **Обновите документацию**, если требуется: docstring (Google Style), README.md, CONTRIBUTING.md.
5. **Добавьте тесты** для новых функций или исправлений.
6. **Отправьте Pull Request** с описанием изменений.

### Требования к коду

- **Стиль кода**: используйте `ruff format` для форматирования и `ruff check` для линтинга.
- **Проверка типов**: код должен проходить проверку `mypy` без ошибок.
- **Тестовое покрытие**: новые функции должны быть покрыты тестами, общее покрытие не должно снижаться.
- **Документация**: 
  - Все публичные функции, методы и классы должны иметь docstring в формате **Google Style** (с разделами `Args`, `Returns`, `Raises`, `Examples`).
  - Примеры в docstring должны быть корректными и предпочтительно в формате doctest.
- **Логирование**: 
  - Используйте модуль `logging` с логгером `logging.getLogger(__name__)`.
  - **Все сообщения логов должны быть на русском языке** (включая debug, info, warning, error).
  - Формат логов: `%(asctime)s - %(name)s - %(levelname)s - %(message)s`, кодировка UTF-8.
- **Исключения**: 
  - Все пользовательские исключения должны наследоваться от базового класса `lifecycle.LifeCycleError`.
  - Сообщения исключений пишите **только на русском языке**, с контекстом (например, `raise ValueError(f"Некорректный возраст: {age}")`).
  - Используйте максимально конкретные встроенные типы исключений (ValueError, TypeError, KeyError и т.д.) или кастомные для бизнес-логики.
  - Избегайте голых `except:` или `except Exception:`. Ловите только конкретные исключения.
- **Версия Python**: проект поддерживает **Python ≥ 3.10**. Используйте актуальный синтаксис (match-case, walrus operator и т.д.).

### Локальная разработка

```bash
# Клонирование вашего форка
git clone https://github.com/ваш-username/lifecycle.git
cd lifecycle

# Установка зависимостей для разработки (используйте uv или pip)
uv sync

# Запуск тестов
make test

# Проверка стиля кода
make lint

# Проверка типов
uv run mypy src/lifecycle

# Автоформатирование
make format
```

### Структура изменений

- **Новая функциональность**: добавьте тесты, документацию и, при необходимости, новые кастомные исключения (с русскими сообщениями).
- **Исправление ошибки**: добавьте тест, воспроизводящий ошибку, если возможно. Убедитесь, что сообщение исключения на русском.
- **Рефакторинг**: убедитесь, что существующие тесты проходят, а логи и исключения не потеряли русский язык.

## Исключения (иерархия и соглашения)

В проекте принята следующая иерархия кастомных исключений (все – с русскими сообщениями):

- `LifeCycleError` – базовый класс для всех исключений библиотеки.
  - `LifeCycleConfigurationError` – ошибка конфигурации (неверные параметры, отсутствующие настройки).
  - `LifeCycleDependencyError` – ошибка зависимостей (циклические, неизвестные).
  - `LifeCycleStateError` – ошибка перехода между состояниями (недопустимое состояние для операции).
  - `LifeCycleRuntimeError` – прочие ошибки времени выполнения.

Если вы добавляете новое исключение, обязательно унаследуйте его от подходящего родительского класса и напишите сообщение на русском.

## Логирование (дополнительные указания)

- Для настройки логирования в приложениях, использующих библиотеку, рекомендуем добавлять:
  ```python
  import logging

  logging.basicConfig(
      level=logging.INFO,
      format="%(asctime)s - %(name)s - %(levelname)s - %(message)s",
      encoding="utf-8",
  )
  ```
- В коде библиотеки не вызывайте `basicConfig` – это ответственность конечного приложения.
- Пример правильного логирования на русском:
  ```python
  logger.info(f"Загружен объект с id={obj_id}")
  logger.error(f"Не удалось сохранить данные: {e}")
  ```

## Окружение разработки

Проект использует:
- **Python ≥ 3.10**
- **uv** для управления зависимостями (допустим pip)
- **pytest** для тестирования
- **ruff** для линтинга и форматирования
- **mypy** для проверки типов
- **Sphinx** для документации (вся документация в коде – Google Style)

Все команды для разработки собраны в `Makefile` (см. `make help`).
